# AQUA Agent Development Guidelines

## Overview
This document guides VS Code agents in implementing features and fixes for AQUA with maximum code reuse, minimal duplication, and adherence to established patterns.

---

## 1. Core Architecture Principles

### 1.1 Composition Over Inheritance
- Use **mixins** for shared functionality (e.g., `InstallMixin`, `CatalogMixin`, `FilesMixin`)
- Delegate to specialized helper classes rather than inheriting
- Example: `Fixer` delegates to `FixerConfigure`, `FixerOperator`, `FixerDataModel`

### 1.2 Singleton Pattern for State Management
- `Reader` maintains a singleton instance (`Reader.instance`) for accessor integration
- Use when you need global state without polluting namespace
- Store latest instance for cross-module access

### 1.3 Logger-Based Cross-Cutting Concerns
- **Always initialize**: `self.logger = log_configure(log_level=loglevel, log_name="ClassName")`
- Pass `loglevel` through constructor for all major classes
- Use in all modules: `from aqua.core.logger import log_configure, log_history`
- Never use `print()` for user-facing output

## 2. Code Reuse Strategies

### 2.1 Configuration Management
**Pattern**: Load once, use many times
```
from aqua.core.configurer import ConfigPath

configurer = ConfigPath(catalog=catalog, loglevel=loglevel)
config = configurer.deliver_intake_catalog(model, exp, source)
```
- Never hardcode paths or defaults
- Always respect `ConfigPath` for machine-aware configuration
- Leverage cached configs for multiple operations

### 2.2 Utility Functions (aqua/core/util/util.py)
**Before writing new helper code, check**:
- `to_list()` - Convert any type to list (preserves None, handles None → [])
- `get_arg()` - Safe argument extraction with defaults
- `extract_attrs()` - Extract attributes from datasets or lists
- `expand_env_vars()` - Recursive environment variable expansion
- `HiddenPrints` - Context manager for suppressing print output
- `username()` - Get current user safely

### 2.3 Data Model Transformations
**Pattern**: Delegate coordinate transformation to `DataModel`
```
from aqua.core.data_model import DataModel

datamodel = DataModel(name="aqua", loglevel=loglevel)
data = datamodel.apply(data, flip_coords=True)
```
- All coordinate standardization goes through this class
- Never hardcode coordinate names—query DataModel config
- Reuse for all xarray datasets

### 2.4 Specialized Processing Chains
- `Fixer` handles variable naming, units, coordinate adjustments
- `Regridder` handles grid transformations
- `FldStat` handles field statistics
- `Streaming` handles temporal data access
- **Never reimplement these**—compose them in Reader methods

### 2.5 Accessor Pattern for xarray Integration
- AQUA provides `.aqua` accessor on xarray objects
- Use `Reader.instance` to access latest reader context
- Reduces method signatures and promotes discoverability

## 3. Pattern Library

### 3.1 Creating New Processing Modules
If adding a new feature, follow this template:

```python
"""Module description"""

from aqua.core.logger import log_configure
from aqua.core.util import to_list, extract_attrs

class MyProcessor:
    def __init__(self, config_name=None, loglevel="WARNING"):
        """
        Args:
            config_name: Configuration identifier
            loglevel: Logging level
        """
        self.logger = log_configure(log_level=loglevel, log_name="MyProcessor")
        self.loglevel = loglevel
        
        # Delegate configuration to specialized class
        self.config_handler = MyProcessorConfig(config_name, loglevel)
    
    def process(self, data, **kwargs):
        """Main processing method"""
        # Use existing utilities
        items = to_list(kwargs.get('items', None))
        
        # Log operations
        self.logger.info("Processing %d items", len(items))
        
        # Perform work
        return data
```

### 3.2 Extending Reader with New Methods
Instead of creating standalone processors:
```python
# GOOD: Add to Reader as mixin
class ProcessorMixin:
    def new_processing(self, variable, **kwargs):
        # Reuse Reader's data, configs, logger
        self.logger.info("Processing %s", variable)
```

### 3.3 CLI Integration Pattern
```python
# commands go in aqua/core/console/
def my_execute(args):
    """Execute from CLI"""
    processor = MyProcessor(loglevel=args.loglevel if hasattr(args, 'loglevel') else 'WARNING')
    processor.process(args.input)

# Register in AquaConsole.command_map
```

## 4. Efficient Data Handling

### 4.1 xarray Best Practices
```python
# Set once at module level
import xarray as xr
xr.set_options(keep_attrs=True)

# Minimize copies
data = data.assign_coords(...)  # Lazy operations
data = data.sel(...)  # Lazy selection
data = data.where(...)  # Lazy masking

# Avoid rechunking—pass chunks parameter to Reader
```

### 4.2 Memory Optimization
- Use `chunks` parameter in `Reader` constructor for dask access
- Streaming is supported—use `streaming=True` for large datasets
- Lazy operations with xarray reduce memory footprint

### 4.3 Reuse Sample Data
```python
# Reader caches sample data
if self.sample_data is None:
    self.sample_data = self.retrieve_plain(...)
    
# Avoid multiple retrievals
```

## 7. Performance Considerations

### 7.1 Lazy Evaluation Chain
Build transformations without executing until needed:
```python
# Chain operations—all lazy until compute
data = reader.retrieve_plain(...)
data = datamodel.apply(data)
data = regridder.regrid(data)
# Only computed when accessed or explicitly called
result = data.compute()
```

### 7.2 Cache Expensive Computations
```python
# For expensive operations (e.g., area calculation)
if not hasattr(self, '_cached_areas'):
    self._cached_areas = compute_areas(...)
return self._cached_areas
```

### 7.3 Parallel Processing
- Use `nproc` parameter for multi-processing operations
- Default: `nproc=4` (configurable via Reader)
- Particularly effective for weights generation

## 8. Common Mistakes to Avoid

### ❌ DON'T
```python
# Don't create new loggers—use instance loggers
logger = logging.getLogger(__name__)

# Don't hardcode paths
data_path = "/path/to/data"

# Don't reimplement data model logic
data.lat = data.lat[::-1]  # This is in DataModel!

# Don't duplicate configuration parsing
fdb_path = metadata['fdb_path']  # Use ConfigPath

# Don't use bare print()
print("Result:", data)  # Use self.logger.info()

# Don't create nested deep copies
data_copy = data.copy(deep=True)  # Use xarray's defaults

# Don't reinvent list handling
if var:
    vars = [var]  # Use to_list(var)

# Don't store state in module globals
GLOBAL_READER = None  # Use singleton pattern
```

### ✅ DO
```python
# Use existing logger from parent
self.logger.info("Processing...")

# Use ConfigPath for all paths
configurer = ConfigPath(catalog=catalog)
data_path = configurer.configdir

# Delegate data model transformations
data = DataModel("aqua").apply(data)

# Load configuration through dedicated class
metadata = configurer.get_metadata()

# Use instance logger
self.logger.info("Result: %s", data)

# Use xarray's defaults (lazy copies)
data = data.assign_coords(new_coords)

# Use utility functions
vars = to_list(var)

# Use singleton pattern
Reader.instance.logger.info(...)
```

### 9.1 Core Dependencies to Leverage
- `xarray` (xr) - Data handling

### 9.2 AQUA Internal Modules
Always prefer existing AQUA modules:
```python
from aqua.core.logger import log_configure, log_history
from aqua.core.util import to_list, expand_env_vars
from aqua.core.configurer import ConfigPath
from aqua.core.data_model import DataModel
```

### 9.3 Avoid External Dependencies
Do not add new external dependencies without discussion.
Check if functionality exists in: numpy, xarray, pandas, or AQUA utils.

## 10. Documentation & Naming

### 10.1 Docstring Format
```python
def method(self, arg1, arg2=None, **kwargs):
    """
    Brief one-liner description.
    
    Longer description if needed, explaining purpose and behavior.
    
    Args:
        arg1 (type): Description
        arg2 (type, optional): Description. Defaults to None.
        **kwargs: Additional options (detail each known key)
        
    Returns:
        type: Description
        
    Raises:
        ExceptionType: When this occurs
        
    Examples:
        >>> obj.method("input", arg2=True)
        desired_output
    """
```

### 10.2 Naming Conventions
- Classes: `PascalCase` (e.g., `Reader`, `Fixer`, `DataModel`)
- Methods/functions: `snake_case` (e.g., `retrieve_plain()`, `apply()`)
- Constants: `UPPER_SNAKE_CASE` (e.g., `DEFAULT_DELTAT`)
- Private/internal: `_leading_underscore`
- Configuration keys: `snake_case`

### 10.3 Variable Naming for Clarity
```python
# Prefer descriptive names that reveal configuration state
src_datamodel = "source"  # source data model
tgt_datamodel = "target"  # target data model
tgt_units = "target units"
fixer_src_units = "fixer source units"
fixer_tgt_units = "fixer target units"

# Use convention prefixes
grid_area = area  # input grid
src_grid_area = source grid area
tgt_grid_area = target grid area
```

## 11. Integration Checklist

When adding a new feature, verify:

- [ ] Logger initialized with `log_configure()`
- [ ] Configuration accessed via `ConfigPath` or dedicated config class
- [ ] Uses `to_list()` for variable-length inputs
- [ ] Reuses existing `DataModel` for coordinate transformations
- [ ] Delegates to specialized classes (not monolithic methods)
- [ ] Attributes preserved with `xr.set_options(keep_attrs=True)`
- [ ] Lazy evaluation used where possible
- [ ] No hardcoded paths, thresholds, or defaults
- [ ] Tests use existing test data in `aqua-data/`
- [ ] Docstrings include examples
- [ ] No external dependencies added

## 13. Examples: Refactoring Inefficient Code

### Example 1: Redundant Logger Initialization
```python
# BEFORE (inefficient)
class OldModule:
    def __init__(self, loglevel="WARNING"):
        self.logger = logging.getLogger(__name__)
        self.logger.setLevel(loglevel)
        
# AFTER (efficient)
class NewModule:
    def __init__(self, loglevel="WARNING"):
        self.logger = log_configure(log_level=loglevel, log_name="NewModule")
```

### Example 2: Configuration Duplication
```python
# BEFORE (inefficient)
def read_fdb_path(catalog):
    import yaml
    with open(f"config/{catalog}/metadata.yaml") as f:
        config = yaml.safe_load(f)
    return config['fdb_path']

# AFTER (efficient)
def read_fdb_path(catalog, loglevel="WARNING"):
    configurer = ConfigPath(catalog=catalog, loglevel=loglevel)
    return configurer.get_metadata()['fdb_path']
```

### Example 3: Composition Over Inheritance
```python
# BEFORE (inefficient—deep inheritance)
class AdvancedReader(BaseReader, ConfigMixin, LoggingMixin):
    pass

# AFTER (efficient—composition)
class AdvancedReader:
    def __init__(self, config_name, loglevel):
        self.config = ConfigPath(catalog=config_name)
        self.logger = log_configure(log_level=loglevel, log_name="AdvancedReader")
```

## 14. Performance Benchmarks & Guidelines

Target metrics for efficient code:
- **Lazy evaluation**: All xarray operations should be lazy until `.compute()`
- **Cache reuse**: Expensive computations cached (areas, weights)
- **Single parsing**: Configuration loaded once, reused multiple times
- **No deep copies**: Avoid `copy(deep=True)` except when necessary
- **Streaming**: For files >1GB, use streaming mode
